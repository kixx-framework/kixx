Typography

One monospace family, system-provided. Hierarchy from size and weight alone.

Typography is the primary design element in this theme — and very nearly the only one. A single monospace family carries body text, headings, labels, and UI copy alike. Hierarchy is set by scale and weight, never color or decoration, so the system stays small, honest, and unmistakably a technical tool.

System fonts — no web fonts at all

The theme downloads zero text font files. It renders in the operating system’s own monospace face — SF Mono, Cascadia Code, DejaVu Sans Mono, or whatever the reader already has — resolved through the --font-mono stack. That is the aesthetic’s core promise made concrete: the page requests no fonts, so it paints instantly and reads the same on any machine.

Type as grid material

Because every glyph is the same width, characters double as units of measure. Reading widths are set in ch (--measure-prose is 72ch), so a column is literally a count of characters, and the whole page aligns to an invisible sheet of graph paper.

Two families of role still exist, but they are weights and sizes of the one face, not different typefaces:

  • Structural headings — h1–h4, bold and larger.
  • Reading & UI text — from .type-body-lg down to .type-caption.
  • Label / overline — uppercase and tracked; the signature for small functional text. Buttons, field labels, and the nav title all use it.
  • Code — the same face with a faint ground, for inline tokens like --text-h1.

Why every size is a fixed rem

Marketing sites use fluid clamp() headings so a hero looks huge on desktop and still fits a phone. An admin panel has no heroes: headings are structural labels inside a capped character column. So we use fixed rem steps — monospace runs wide, so the steps sit a little lower than a proportional scale would:

  • Predictable density. A section title is the same size every time, so dense screens and data tables stay stable instead of breathing with the viewport.
  • Zoom & accessibility. Fixed rem scales cleanly with browser zoom; vw-based fluid type partially ignores it, working against WCAG 1.4.4 (Resize Text).
  • Simplicity. If a heading is ever too large on mobile, add one discrete breakpoint step — never continuous scaling.
*

Rule of thumb

Line-height is role-dependent, never one global value. Big headings run tight (1.2), reading copy opens to 1.65 for breathability, and dense UI text sits near 1.45 to save vertical space.

Guidelines

Do

  • Reach for semantic elements first — h1–h4 and p render correctly with no classes.
  • Signal importance with size and weight, not color, italics, or underlines.
  • Let paragraphs breathe: .type-body-lg at 1.65 for anything people actually read.
  • Keep a single h1 per screen as the page title.

Don’t

  • Introduce a second typeface or download a web font — the system is one system-provided mono family.
  • Pick a heading level for its size — choose the level for structure, then restyle with a .type-* class.
  • Invent new font sizes; every size lives in a --text-* token.
  • Reintroduce fluid clamp() type — fixed rem only, with a breakpoint step if needed.
