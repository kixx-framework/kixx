Style Guide

The working reference for placeholder UI design for a default Kixx site.

This style guide is a place-holder as part of your default Kixx installation. The intention is for you to replace this style guide once you have established the aesthetic for your site. It documents the visual system, reusable layout primitives, and component patterns used on the public facing site as well as inside the authenticated admin experience; it is not a marketing guide, brand manual, or general framework documentation.

Use the style guide when creating or changing public site or admin pages, forms, screens, dashboards, and other operational UI. The guide keeps UI work aligned around the same tone — Hypertext Minimalism: fast, plain, highly structured, and honest. Performance through omission.

What it contains

The sections move from design intent to concrete implementation details.

Foundations

Aesthetic, Typography, and Colors define the admin panel's tone, hierarchy, token system, and theme-aware color contract.

Layout

Layout documents the composable primitives used to arrange admin screens: flow, cluster, grid, switcher, sidebar, and center.

Components

Buttons, Cards, Text Fields, Multi-Line Text Areas, and Callouts show live examples, usage rules, states, and accessibility guidance.

How to use it

Start with the Aesthetic when you need the design direction behind the system. Then move to the specific foundation or component page for the thing you are building, and copy the documented markup patterns instead of inventing a parallel class vocabulary.

For implementation work, treat each page as a decision checklist:

  • Use the documented semantic tokens before adding new colors, spacing, typography, borders, or states.
  • Reach for layout primitives before writing custom flex or grid rules.
  • Match component anatomy, variants, states, and usage examples before adding modifiers.

If a pattern is missing, prefer extending the shared design system with a reusable token, primitive, or component rather than solving one page with a one-off style. This guide should remain the source of truth for how the UI is built.

Rule of thumb

Build admin UI from the guide outward: foundations first, layout second, component details last. The result should feel like one system even when many pages are built at different times.
