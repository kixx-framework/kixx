Style Guide

The working frontend reference for Kixx pages, templates, and shared CSS.

Style Guide Place-Holder

This style guide is a place-holder as part of your default Kixx installation. The intention is for you to replace it once you have established the aesthetic for your site.

Use this style guide before changing any frontend markup or CSS. It is the live source of truth for the site’s aesthetic, design tokens, layout primitives, type roles, components, form controls, and documentation patterns.

The system is intentionally narrow: Hypertext Minimalism. Build from semantic HTML, system monospace type, thin structural rules, square controls, explicit links, functional whitespace, and color used only as a signal. When a page feels over-designed, reduce the surface area before adding another style.

For implementation details, pair this guide with docs/frontend-development-guide.md. That document explains the no-inline-styles rule, the stylesheet organization, BEM naming, token layering, layout primitive contract, and page-local stylesheet pattern.

Development workflow

Follow this order when doing frontend work.

1. Read the intent

Start with Aesthetic. It defines the tone behind every later choice: fast, plain, structured, document-like, and honest.

2. Reuse foundations

Use Typography, Colors, and Layout before introducing new type, color, spacing, or arrangement rules.

3. Copy the pattern

For Buttons, Cards, Text Fields, Multi-Line Text Areas, and Callouts, copy the documented anatomy and states instead of inventing parallel markup.

How to use it

Start with the Aesthetic when you need the design direction behind the system. Then move to the specific foundation or component page for the thing you are building, and copy the documented markup patterns instead of inventing a parallel class vocabulary.

For implementation work, treat each page as a decision checklist:

  • Use the documented semantic tokens before adding new colors, spacing, typography, borders, or states.
  • Reach for layout primitives before writing custom flex or grid rules.
  • Match component anatomy, variants, states, and usage examples before adding modifiers.

If a pattern is missing, prefer extending the shared design system with a reusable token, primitive, or component rather than solving one page with a one-off style. This guide should remain the source of truth for how the UI is built.

Decision rules

Do

  • Use semantic HTML first; add classes only when the design system needs them.
  • Resolve colors, spacing, type, borders, and radii through existing tokens.
  • Compose layout with primitives before writing bespoke flex or grid CSS.
  • Prefer existing component anatomy, variants, and states over one-off selectors.
  • Use page-local page_stylesheet includes only for styling that truly belongs to one page.

Avoid

  • Inline style attributes in templates.
  • Raw color literals or direct --palette-* references inside component rules.
  • New wrappers, modifiers, or utilities when a primitive or component already fits.
  • Decorative gradients, shadows, rounded corners, icons, images, or motion.
  • Adding dependencies, web fonts, or icon libraries for visual polish.
*

Rule of thumb

If you need new CSS, first decide whether it is a token, a primitive, a shared component, a form control, or a page-local exception. Then edit the matching file and update the style guide example when the pattern becomes reusable.
