Admin Style Guide

The working reference for building consistent Kixx admin panel screens.

This style guide is for Kixx admin panel development only. It documents the visual system, reusable layout primitives, and component patterns used inside the authenticated admin experience; it is not a public marketing guide, brand manual, or general framework documentation.

Use it when creating or changing admin pages, forms, settings screens, dashboards, and other operational UI. The guide keeps admin work aligned around the same tone: clear, welcoming, highly structured, and practical for repeated use.

What it contains

The sections move from design intent to concrete implementation details.

Foundations

Aesthetic, Typography, and Colors define the admin panel's tone, hierarchy, token system, and theme-aware color contract.

Layout

Layout documents the composable primitives used to arrange admin screens: flow, cluster, grid, switcher, sidebar, and center.

Components

Buttons, Cards, Text Fields, Multi-Line Text Areas, and Callouts show live examples, usage rules, states, and accessibility guidance.

How to use it

Start with Aesthetic when you need the design direction behind the system. Then move to the specific foundation or component page for the thing you are building, and copy the documented markup patterns instead of inventing a parallel class vocabulary.

For implementation work, treat each page as a decision checklist:

  • Use the documented semantic tokens before adding new colors, spacing, typography, borders, or states.
  • Reach for layout primitives before writing custom flex or grid rules.
  • Match component anatomy, variants, states, and usage examples before adding modifiers.
  • Read the accessibility and guidelines sections before shipping a new screen.

If a pattern is missing, prefer extending the shared admin system with a reusable token, primitive, or component rather than solving one page with a one-off style. The guide should stay the source of truth for how admin panel UI is built.

Rule of thumb

Build admin UI from the guide outward: foundations first, layout second, component details last. The result should feel like one system even when many pages are built at different times.
