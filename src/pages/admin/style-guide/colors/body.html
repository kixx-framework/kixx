Colors

Effective use of colors in the Kixx Admin Panel theme.

Color is a three-tier token system. Each tier has one job, and references flow in a single direction — so a value changes in exactly one place, and theming is something the system does, not something a component worries about.

  • Reference palette (--palette-*) — the raw, theme-agnostic ingredients: five saturated hues, black, white, and a neutral grey ramp. The only place a literal color value lives.
  • Semantic (--color-*) — role- and theme-aware tokens resolved from the palette: --color-ink, --color-bg, --color-rule, --color-ink-primary. This is the layer you build with.
  • Component (e.g. --color-button-bg-primary, --card-accent) — scoped tokens built from the semantic layer for one component.

The single direction is what makes it a contract rather than a convention: component → semantic → reference. The light-dark() theming all happens in the semantic tier, so a component written against --color-rule is correct in both themes for free.

The rule

A component rule consumes a semantic --color-* token only. It never names a raw --palette-* token or writes an hsl() / hex literal. If you reach for the palette in a component, the role you actually want is a missing semantic token — add it.

Status colors

A functional family, not brand accents. They sit beside the accents in the semantic tier but carry a separate job — signalling state — so “something is wrong” never shares a color with “primary action.” Two members — both read as a foreground mark (text, border, icon) and both theme-aware so they hold contrast on light and dark: --color-status-danger (deep red on white, brighter on black — as .field--error uses it) and --color-status-warning (a deep amber on light surfaces, the brand yellow on dark — brand yellow alone is too light to read on white). For a yellow fill, reach for --palette-yellow directly with text in --color-ink-on-accent, like the tertiary button — that's decoration, not a signal.

Guidelines

Do

  • Style components with semantic --color-* tokens — --color-ink, --color-rule, --color-bg.
  • When a role is missing, add a semantic token (as --color-ink-on-accent was) instead of hardcoding.
  • Let light-dark() in the semantic tier handle theming; keep components theme-agnostic.
  • Carry brand color in accents — link underline, the primary button — not in body text. (The focus ring is its own blue signal, not a brand accent.)

Don't

  • Name a raw --palette-* token or write an hsl() / hex literal in a component rule.
  • Reach past the semantic layer — not even “just this once.”
  • Encode a state as a new literal; add a palette shade plus a component token.
  • Lean on color alone to carry meaning — pair it with weight, an icon, or text.
