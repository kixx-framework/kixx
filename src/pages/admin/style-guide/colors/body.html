Colors

High-contrast monochrome, with color reserved for a single functional signal.

The base scheme is dark-gray-and-off-white — a faintly warm paper and a near-black ink that read like a printed spec sheet, not a glaring screen. Color is a signal, not a mood: one accent for a link, and a small functional family for status. Nothing else is tinted.

It is still a three-tier token system. Each tier has one job, and references flow in a single direction — so a value changes in exactly one place, and theming is something the system does, not something a component worries about.

  • Reference palette (--palette-*) — the raw, theme-agnostic ingredients: a neutral ramp for each theme, a hairline ramp, and the signal hues. The only place a literal color value lives.
  • Semantic (--color-*) — role- and theme-aware tokens resolved from the palette: --color-ink, --color-bg, --color-rule, --color-ink-link. This is the layer you build with.
  • Component (e.g. --color-field-border, --card-accent) — scoped tokens built from the semantic layer for one component.

The single direction is what makes it a contract: component → semantic → reference. The light-dark() theming all happens in the semantic tier, so a component written against --color-rule is correct in both themes for free.

*

The rule

A component rule consumes a semantic --color-* token only. It never names a raw --palette-* token or writes an hsl() / hex literal. If you reach for the palette in a component, the role you actually want is a missing semantic token — add it.

*

Status colors

A functional family, kept off the link accent so “something is wrong” never shares a color with “this is a link.” All three read as a foreground mark (text, border, icon) and are theme-aware so they hold contrast on light and dark: --color-status-danger (a deep red on paper, brighter on the void — as .field--error uses it), --color-status-warning (a deep amber on light, brand yellow on dark), and --color-status-success for confirmations. Because these are tuned for marks (3:1), not body text (4.5:1), keep the message itself in --color-ink.

Guidelines

Do

  • Style components with semantic --color-* tokens — --color-ink, --color-rule, --color-bg.
  • When a role is missing, add a semantic token instead of hardcoding.
  • Let light-dark() in the semantic tier handle theming; keep components theme-agnostic.
  • Reserve color for signals: the link accent and the status family. Everything else is ink on paper.

Don’t

  • Name a raw --palette-* token or write an hsl() / hex literal in a component rule.
  • Reach past the semantic layer — not even “just this once.”
  • Tint a surface, border, or control for decoration — color must earn its place as a signal.
  • Lean on color alone to carry meaning — pair it with weight, a mark, or text.
