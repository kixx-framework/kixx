Forms

Assembling fields, spacing, feedback, and submits into one working flow.

A form is not its own component. It is a composition of primitives: stack fields in a .flow, give that flow the form rhythm with .field-stack, and sit the submit in a .cluster. On a standalone form page, cap the column with .center--form. The form-specific decision is rhythm, not a new wrapper vocabulary.

Fields need more air between them than the default .flow gap gives. .field-stack binds that spacing to --form-field-gap, so every form can move together when the rhythm is retuned.

Anatomy

A stack of fields with a generous gap, closed by a submit in a cluster. The form element is the flow container; its direct children are the rows.

We’ll only use this to sign you in.

A live layout — tab through the fields to the button. The submit is inert here.

  1. Form flow — <form class="flow field-stack">. The form is the flow container; each field, callout, and submit cluster is a direct child spaced by the flow gap.
  2. Field — a standard .field with label, control, optional hint, and optional error text.
  3. Field gap — the rhythm between rows, set by .field-stack to --form-field-gap so fields never crowd each other.
  4. Submit — the primary action in a .cluster, so secondary actions can sit beside it and wrap cleanly on narrow screens.

Field Rhythm

Compose .field-stack onto a .flow form to swap the default gap for the roomier form gap. Same fields, different breathing room.

.flowDefault gap — too tight for a form
.flow.field-stackForm gap via --form-field-gap

.field-stack only rebinds the flow gap:

.field-stack {
--flow-space: var(--form-field-gap);
}

--form-field-gap sits intentionally between --space-lg and --space-xl. It is roomier than ordinary document flow, but still part of the same measured, utilitarian rhythm.

Hints and Errors

Standing guidance and validation feedback are two different messages, so they are two different elements.

At least 8 characters, including a number.

.field__hintPersistent guidance, muted

At least 8 characters, including a number.

Password must be at least 8 characters.

.field__hint + .field__errorGuidance and correction, together

Put persistent advice in .field__hint; it stays muted even beside an error. Put corrective validation feedback in .field__error; it carries the danger color and role="alert" when rendered after a failed submit.

Add .field--error to the wrapper to turn the control border red, set aria-invalid="true" on the control, and reference every rendered message id in aria-describedby. When a field has no standing guidance, render .field__error alone.

Standalone Form Pages

A form that is the page — login, signup, settings — sits in a centered column capped to the form measure so its lines never stretch too wide to scan.

Wrap the page content in .center.center--form.flow: .center--form caps the column to --measure-form, and the outer .flow spaces the header, the form, and any trailing links. Keep .field-stack on the inner <form>, not on the wrapper. Page rhythm and field rhythm are different jobs.

A form embedded in a wider page can skip .center--form, but should still use .field-stack so its fields breathe like every other form.

Usage

A standalone form page. The wrapper owns the column and page rhythm; the form owns the field rhythm.

<div class="center center--form flow">
<header class="admin-content-section__header">
<h2>Log in</h2>
<p>Sign in to the admin panel.</p>
</header>

<form class="flow field-stack" method="post" action="/login" novalidate>
<div class="field field--error">
<label class="field__label" for="email">Email address</label>
<input class="field__input" id="email" name="email" type="email"
aria-describedby="email-hint email-error" aria-invalid="true">
<p class="field__hint" id="email-hint">Use the email from your invite.</p>
<p class="field__error" id="email-error" role="alert">Enter a valid email address.</p>
</div>

<div class="field">
<label class="field__label" for="password">Password</label>
<input class="field__input" id="password" name="password" type="password">
</div>

<div class="cluster">
<button class="button button--primary" type="submit">Log in</button>
</div>
</form>
</div>

Accessibility

  • Every control keeps a visible <label> tied with for/id. Never use placeholder text as the only label.
  • Put a form-level failure in a callout with role="alert" as a direct child of the form flow, above the fields.
  • Put field-level errors in a dedicated .field__error with role="alert", beside any persistent .field__hint.
  • Reference every rendered hint and error message with aria-describedby, and mark invalid controls with aria-invalid="true".
  • Let the submit be a real type="submit" button so the form posts on Enter from any field.

Guidelines

Do

  • Put .field-stack on the <form> so every form shares one rhythm.
  • Stack fields in a single column; the eye moves straight down and never hunts across a row.
  • Group submit and secondary actions in a .cluster so they wrap gracefully.
  • Keep standing guidance in .field__hint and validation feedback in .field__error.

Don’t

  • Leave a form on the bare .flow default when it is collecting multiple fields.
  • Hard-code a one-off gap with inline styles or page-local overrides; retune --form-field-gap instead.
  • Put .field-stack on the outer .center--form wrapper; it would widen header and footer gaps too.
  • Use .field__hint as the error-message slot; render .field__error for validation feedback.
