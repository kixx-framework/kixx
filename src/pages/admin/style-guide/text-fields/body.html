Text Fields

Single-line inputs for names, emails, numbers, and other short values.

Text fields collect short, structured input. They are intentionally plain — a clear border, a recessed fill, and nothing that distracts from the data being entered. The label sits above in the label register (mono, uppercase), and optional hint text below carries context without adding visual weight.

When to use: a text field is for a single line of user input — names, email addresses, numbers, URLs, and similar. When the user needs to write more than a line or two, reach for a multi-line text area instead. When choosing from a fixed set of options, reach for a select or radio group.

Anatomy

Label above, input in the middle, hint below — three parts in a gap-based column stack.

This is how your name appears to other users.

  1. Label — mono, uppercase, muted; sits directly above the input and is always connected with a matching for/id pair.
  2. Input — the interactive control; a 1px muted box on the page surface, square corners, and a 2.75rem minimum height for a comfortable target.
  3. Hint — optional caption below; describes the expected format or explains a constraint. When the field is in error, the hint becomes the error message.

Variants

The field wrapper has one class modifier, .field--error. Read-only and disabled are HTML attributes, not class modifiers.

States

Click into the live field, or press Tab, to see focus; hover to see the border activate.

This is a live, interactive field.

Default
On the page surface, 1px muted box. Quiet until the user engages.
Hover
Border intensifies to full --color-ink — the field sharpens without moving or jumping.
Focus
Border stays active and a 2px --color-focus-outline ring appears, offset 2px from the border edge.
Disabled
Muted border and muted text at half opacity; cursor becomes not-allowed. Hover and focus are suppressed.
Read-only
Border becomes dashed — a visual signal the value is visible but not editable. The field remains focusable and its text is selectable.
Error
Border turns --color-field-border-error (the danger red, from the status color). The hint below carries the error message in the same color.

Usage

Always pair every input with a visible <label>. Use aria-describedby to link hint and error text to the input.

<!-- Standard text field -->
<div class="field">
<label class="field__label" for="name">Display name</label>
<input class="field__input" id="name" type="text"
placeholder="e.g. Jane Smith">
<p class="field__hint">How your name appears to other users.</p>
</div>

<!-- Error state -->
<div class="field field--error">
<label class="field__label" for="email">Email address</label>
<input class="field__input" id="email" type="email"
aria-describedby="email-hint" aria-invalid="true">
<p class="field__hint" id="email-hint">Enter a valid email address.</p>
</div>

<!-- Read-only value (selectable, not editable) -->
<div class="field">
<label class="field__label" for="account-id">Account ID</label>
<input class="field__input" id="account-id" type="text"
value="usr_01HZ3F9JK7" readonly>
</div>

<!-- Disabled (unavailable in context) -->
<div class="field">
<label class="field__label" for="api-key">API key</label>
<input class="field__input" id="api-key" type="text"
value="sk_live_••••••••" disabled>
<p class="field__hint">Upgrade to enable API access.</p>
</div>

Accessibility

  • Always connect a <label> to its input with a matching for/id pair — never use placeholder text as the only label.
  • Link hint and error text to the input with aria-describedby so screen readers announce it when the field receives focus.
  • Mark invalid fields with aria-invalid="true"; remove the attribute (or set it to false) once the value is corrected.
  • Prefer disabled for fields the user cannot interact with at all; use readonly for values the user can copy but not change.
  • Do not rely on the error border alone to signal a problem — color never carries meaning by itself. The hint text is the accessible error message.
  • The 2.75rem minimum height meets touch-target guidelines on mobile; keep it.

Guidelines

Do

  • Stack fields vertically in a form so the eye moves straight down the page and never hunts across a row for a matching label.
  • Use hint text to explain a constraint — minimum length, expected format — before the user makes an error.
  • Use the right type attribute (email, tel, number, url, password): it enables the right mobile keyboard and unlocks browser autofill.
  • Size fields to their expected content, not the full column width.

Don't

  • Use placeholder text as a label — it disappears the moment the user starts typing.
  • Validate on every keystroke; validate on blur or on submit.
  • Remove the focus ring from inputs — keyboard users depend on it.
  • Use disabled when readonly is more appropriate — disabled fields are completely ignored by assistive technology.
