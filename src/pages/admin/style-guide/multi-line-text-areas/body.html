Multi-Line Text Area Fields

Multi-line inputs for comments, descriptions, and other free-form prose.

A text area is a text field that has grown a second dimension. It reuses the same parts — the mono label above, the recessed input, the hint below — and every state and modifier carries over unchanged. The only thing it adds is room: a taller default, a reading-comfortable line height, and a resize grip the user can drag.

That reuse is deliberate. A <textarea> takes the same .field wrapper and .field__input class as a single-line input; the theme adds just three multi-line rules on top. There's no new vocabulary to learn — .field--error, readonly, and disabled all behave exactly as they do on a text field.

When to use: for input that runs longer than a line and has no fixed structure — a comment, a bio, a description, an internal note. For a single short value, use a text field. For rich formatting or code, reach for a purpose-built editor, not a bare text area.

Anatomy

The same three parts as a text field — label, control, hint — with the control sized for several lines and draggable taller.

A sentence or two helps teammates find it later.

  1. Label — identical to the text field: mono, uppercase, muted, tied to the control with a for/id pair.
  2. Text area — a <textarea> on .field__input; same 1px box on the page surface, but a 6.5rem minimum height and body line-height for reading multiple lines.
  3. Resize grip — the corner handle, constrained to resize: vertical so the area grows downward and never breaks the column width.
  4. Hint — optional caption below; becomes the error message when the field is in error.

States

Identical to the text field, plus one affordance: drag the corner grip to resize. Click in to see focus; hover to see the border activate.

Default
On the page surface, 1px muted box, sized to its rows. Quiet until engaged.
Hover
Border intensifies to full --color-ink — no movement, just a sharper edge.
Focus
Border stays active and a 2px --color-focus-outline ring appears, offset 2px.
Resize
A vertical-only grip in the corner; the area grows downward without disturbing the layout width.
Disabled
Muted border and text at half opacity, not-allowed cursor; hover, focus, and resize are suppressed.
Read-only
Border becomes dashed; the text stays selectable and the field focusable, but it can't be edited.
Error
Border and hint turn the danger red via .field--error, exactly as on a text field.

Usage

Same wrapper and label rules as a text field. Set rows for a predictable starting height; never self-close a <textarea> — its value is its text content.

<!-- Standard text area -->
<div class="field">
<label class="field__label" for="bio">Bio</label>
<textarea class="field__input" id="bio" rows="4"
placeholder="Tell us about yourself…"></textarea>
<p class="field__hint">A sentence or two is plenty.</p>
</div>

<!-- Error state (same modifier as a text field) -->
<div class="field field--error">
<label class="field__label" for="reason">Reason</label>
<textarea class="field__input" id="reason" rows="3"
aria-describedby="reason-hint" aria-invalid="true"></textarea>
<p class="field__hint" id="reason-hint">Please add a little more detail.</p>
</div>

Accessibility

  • Connect the <label> with a matching for/id pair, and link hint and error text with aria-describedby — same as a text field.
  • Mark an invalid area with aria-invalid="true" and remove it once corrected; the hint text, not the color, is the accessible error message.
  • Set the rows attribute so the area opens at a predictable height instead of collapsing to a single line on load.
  • Keep resize: vertical available — pulling the area taller is an accessibility aid for anyone reviewing a long entry. Don't set resize: none.
  • If you cap length with maxlength, show the remaining count in the hint so the limit isn't a surprise at the last character.

Guidelines

Do

  • Size the starting rows to the expected answer — a few lines for a comment, more for a description.
  • Reuse the field vocabulary: .field--error, the hint, the label register all apply unchanged.
  • Let users grow the area; a long entry is easier to review when it's all visible.
  • Use a text area only when the input genuinely runs to multiple lines.

Don't

  • Disable resizing, or allow horizontal resize — it breaks the column and the line length.
  • Use a text area for a single short value; that's a text field.
  • Self-close the tag (<textarea />) — put any initial value between the tags.
  • Let it stretch to an unbounded width; long lines are tiring to read. Keep it within the form column.
